# 설계 원칙 (정본)

대화하며 도출한 하네스/AX 설계 원칙. 흩어진 걸 여기로 모음. 진입 순서 원칙은 [[roadmap]] 참조.

## 좌표 — 나는 지금 어디 있나

하네스 엔지니어링은 두 축으로 본다:

```
             통제(control) ──────────► 자율(autonomy)
   dev-time      [team-harness]          [외부 프로세스 자율]
      │           ↑ 내 현위치
      ▼
   runtime                             [타깃 회사]  ← 목적지
```

- **축 A (통제↔자율)**: 나는 통제(강제·게이트·검증)를 극한까지 팠고, 자율(무인 장기실행)이 성장 모서리.
- **축 B (dev-time↔runtime)**: 나는 개발 하네스를 제도화했고, runtime 비즈니스 하네스가 다음.
- 회사 일 = **대각선 이동**. 통제 근육을 자산으로, 자율+runtime으로.

## 핵심 원칙

1. **모델이 아니라 하네스가 병목** — `Agent = Model + Harness`. 투자는 컨텍스트·도구·가드레일·검증에.

2. **컨텍스트 보존 = 단계마다 격리 + 상태는 디스크로** — 긴 세션 하나가 컨텍스트를 채우게 두지 말고, 단계마다 새 컨텍스트 + 상태 파일 + 체크포인트. 결정론적 오케스트레이터 + 지능 작업(LLM) 분리("반복은 스크립트, 메인은 의도"). **구현은 손으로 재발명 말고 네이티브 프리미티브로:** 인터랙티브/dev → **Workflow + 서브에이전트**, 배포형 자율(cron 무인) → **Agent SDK / Managed Agents**(스케줄 배포·세션 상태·resume).

3. **암묵지 → 기계가 읽는 파일** — 서비스 오너의 암묵지·지표 정의를 에이전트가 읽는 구조화 파일(md/yaml)로. **복잡한 RAG 없이** 고품질. = OKF = semantic layer.

4. **git은 숨은 기판, 표면은 친절하게** — AI 산출물 버전관리의 기판은 거의 무조건 git. 단 비개발자는 git을 *직접* 안 씀. **친절한 표면(시트·챗·버튼) 밑에서 git이 보이지 않게 커밋.** build 타깃 = "git을 가르친다"가 아니라 "친절한 행동 → git 연산 번역 레이어."

5. **git ≠ git-flow — 의식의 무게를 산출물별로 right-size** — 코드엔 풀 flow(브랜치·PR·리뷰·머지). 리포트·지표정의 같은 비-코드 산출물엔 경량 버전+출처+롤백이면 충분. team-harness 강제를 그대로 복붙 금지(역방향 오버엔지니어링).

6. **강제는 prose가 아니라 기계 게이트로** — 규칙은 branch protection·CI·훅으로. CLAUDE.md 강제 조항은 비강제 환상.

7. **사람 확인 게이트 → 런타임 가드레일로 번역** — dev 하네스의 "초안까지만, 사람 확인"을 자율 runtime으로 옮길 때: 어떤 액션이 무인 가능(리포트)이고 어떤 건 사람 필수(발주·발송)인지 + 중단조건·최대반복·액션 샌드박스·승인 큐로 인코딩. 자율 = 사람 제거이므로 이게 진짜 어려운 부분(페이즈 러너는 껍데기).

8. **모호함 없고 패턴 있는 영역만 위임, 최종 판단은 인간** — 신뢰가 자동화를 확장시킨다.

## 내 자산 (과소평가 금지)

team-harness = "사람이 git을 몰라도 엄격한 버전관리 + 순서·결과 보장이 자동으로 일어나게 하는 시스템." 강제(훅·게이트)가 그 핵심. **올바른 프리미티브를 이미 만듦** — 비개발자 AX가 필요로 하는 바로 그것. 대부분은 자율만 알고 통제를 몰라 사고 냄; 나는 그 반대라 자율을 *안전하게* 얹을 수 있음.
